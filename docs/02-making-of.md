# How A Project About Preventing Drift Drifted

This project did not emerge from perfect discipline.

It emerged the way many real projects do: from energy, intuition, momentum, taste, live conversation, opportunistic synthesis, and a string of "yes, and" decisions that were often good locally and only later questionable globally.

That is not a defect in the story. It is the story.

## The Confession

This repository teaches scoping, specing, planning, seams, stopping points, testing, debugging, handoffs, compaction, and tightening. It argues that projects go wrong when they do not keep those things explicit.

And yet this project itself was built, at least initially, by doing a lot of that implicitly.

We did not start with a fully written brief for the repo itself.

We did not lock a spec before building.

We did not begin with a disciplined tracking system for this repository's own execution state.

We did something much more human:

- noticed an interesting shape
- started building toward it
- kept seeing better possibilities
- followed them
- tightened only after enough structure existed to tighten

In other words, we drifted.

Not aimlessly. Not incompetently. But unmistakably.

## Why That Matters

A project like this could easily become dishonest.

It could present best practices as if good projects are born from clean linear execution, as if the people building them always know the right scope early, as if drift is a sign of weakness instead of a normal byproduct of live thinking.

That version would be false.

Real projects drift because reality is discovered unevenly.

The problem is not drift itself. The problem is drift that remains unnamed, unexamined, and uncorrected. The problem is when a project keeps accumulating motion without re-establishing clarity.

That is exactly why this repository needs to exist.

## What Actually Happened Here

This project started as an idea for a self-referential guide: a project about how to create projects.

From there it expanded quickly:

- first into lifecycle docs and templates
- then into a single-source model
- then into a handbook, CLI, and browser app
- then into a tracking kit
- then into branch/worktree support for speculative agent work

Those additions were not all wrong. Many of them were strong.

But they were not all the result of prior explicit scope control.

Some were genuine product discoveries.
Some were good instincts.
Some were us enjoying the making.
Some were premature.
Some only became justified after the fact.

That mixture is common. It is also dangerous if nobody says so.

## The Expected Phenomenon

The deeper truth is that a project about project hygiene should expect this exact failure mode.

Why?

Because this kind of project naturally attracts synthesis.
It invites framework-building.
It rewards pattern recognition.
It makes it easy to keep seeing one more elegant connection.

That means it is unusually vulnerable to:

- scope expansion disguised as coherence
- tooling growth disguised as completeness
- documentation growth disguised as rigor
- polished advice outrunning the discipline used to produce it

That is not irony. It is evidence.

If even this project tends to slide in that direction, then the need it addresses is real.

## What We Want To Model Instead

Not perfection.

Not bureaucratic self-punishment.

Not the fantasy that every worthwhile project begins with a frozen brief and obediently follows it.

What we want to model is a more honest cycle:

1. Explore.
2. Notice drift.
3. Name it plainly.
4. Recover structure.
5. Keep going with better boundaries.

That is the actual discipline.

The purpose of this repository is not to eliminate the generative mess at the beginning of work. It is to help people avoid staying in that mess longer than necessary.

## The Correction

This repository is now trying to do on purpose what it first did only partially:

- use its own brief/spec/plan logic on itself
- use its own tracking layer on itself
- separate speculative branch work from trusted integrated work
- preserve the making-of instead of erasing it

That correction is part of the product, not a cleanup step outside the product.

## The Point

If this document makes the project feel slightly less polished and more human, good.

Projects drift.
Even projects built to prevent drift drift.

That does not discredit the advice.

It proves the advice is about a real problem.

The standard is not "never drift."

The standard is:

drift, notice, tighten, continue.
